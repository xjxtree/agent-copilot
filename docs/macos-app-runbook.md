# macOS App Runbook

> Status: current-stage app run and validation rules. The project currently defines no public distribution artifact. The only app bundle path in current use is `dist/SkillsCopilot.app`.
>
> Future public distribution preparation is tracked separately in [`distribution-runbook.md`](./distribution-runbook.md).

## Shared App Bundle

Both current scenarios use the same bundle path:

`dist/SkillsCopilot.app`

This bundle is regenerated by `script/build_and_run.sh`. It is not signed, notarized, stapled, zipped, or packaged as a public distribution artifact.

## Naming

| Name | Command | Data environment | Purpose |
| --- | --- | --- | --- |
| Local App Run | `./script/build_and_run.sh run` or `pnpm dev:macos` | Real local `HOME`, default app data, real Claude config | Manual local use and visual/effect checks against the developer's actual environment |
| Local App Launch Verify | `./script/build_and_run.sh --verify` or `pnpm build:macos` | Real local `HOME`, default app data, real Claude config | Rebuild `dist/SkillsCopilot.app`, launch it, and verify the process starts |
| Smoke App Run | `pnpm smoke:macos-app -- --fixture-data --capture-window` | Temporary fixture `HOME`, temporary app data, synthetic Claude/Codex/opencode skills/settings | Automated post-development validation that does not touch real user config; fails if `dist/SkillsCopilot.app` is stale |
| macOS Check | `pnpm check:macos` | Runs Local App Launch Verify, then Smoke App Run | Combined local quality gate for fmt/test/clippy/build/launch/smoke/window screenshot |

## When `dist/SkillsCopilot.app` Is Updated

`dist/SkillsCopilot.app` is regenerated whenever `script/build_and_run.sh` runs in `run` or `--verify` mode. The script:

- builds `skills-copilot-service`
- builds the SwiftPM macOS app
- removes and recreates `dist/SkillsCopilot.app`
- copies the Swift app binary into `Contents/MacOS/SkillsCopilot`
- copies the Rust service sidecar into `Contents/Resources/skills-copilot-service`
- copies Swift localized resources from `apps/macos/Sources/SkillsCopilot/Resources` into `Contents/Resources`
- copies `apps/macos/Sources/SkillsCopilot/Resources/AppIcon.icns` into `Contents/Resources/AppIcon.icns`
- writes `Contents/Info.plist`
- launches the app

Therefore these commands update `dist/SkillsCopilot.app` before launching or checking it:

```sh
./script/build_and_run.sh run
./script/build_and_run.sh --verify
pnpm dev
pnpm build
pnpm dev:macos
pnpm build:macos
pnpm check:macos
```

`pnpm smoke:macos-app` does **not** build or package the app. It only validates the existing `dist/SkillsCopilot.app`.

By default, Smoke App Run checks freshness before it launches:

- Swift sources / `apps/macos/Package.swift` must not be newer than `Contents/MacOS/SkillsCopilot`
- Rust workspace sources / Cargo manifests must not be newer than `Contents/Resources/skills-copilot-service`
- Swift localized resources must not be newer than `Contents/Resources/en.lproj/Localizable.strings`
- `apps/macos/Sources/SkillsCopilot/Resources/AppIcon.icns` must not be newer than `Contents/Resources/AppIcon.icns`
- `script/build_and_run.sh` must not be newer than the generated app binary, sidecar, icon, or `Info.plist`

If any input is newer, Smoke App Run fails and tells the developer to run `./script/build_and_run.sh --verify` or `pnpm check:macos`. A diagnostic-only escape hatch exists: `pnpm smoke:macos-app -- --allow-stale-app`, but normal validation must not use it.

## Smoke App Run

Smoke App Run must use fixture data:

```sh
pnpm smoke:macos-app -- --fixture-data --capture-window
```

It verifies:

- `dist/SkillsCopilot.app` exists
- bundle id is `dev.skills-copilot.native`
- `AppIcon.icns` is bundled
- packaged `skills-copilot-service` exists
- a visible `SkillsCopilot` app window exists
- app-window-only screenshot capture works
- service fixture flow covers Scan, Enable/Disable, Settings save, Snapshot Preview, and Snapshot Rollback
- V2.4 opencode fixture roots are isolated to temporary native opencode roots: `~/.config/opencode/skills` and project `.opencode/skills`; smoke must not create or scan real user opencode config
- when the integrated service reports opencode rows from `catalog.scanAll`, smoke verifies global opencode skills without project context, project opencode skills with project context, and `config.toggleSkill` rejection/read-only behavior
- V2.9 tool-global validation is covered by Rust service tests, Swift UI/model tests, layout checks, and service fixtures: tool-global rows must display as read-only previews, toggle must remain disabled, import writes only app-controlled staging/catalog, export manifests remain reproducible, and install/copy to Claude/Codex must be confirmation-gated through verified write paths
- V2.10 execution safety validation is a boundary check, not a runner check: smoke, scan, import, export, install, LLM prepare, state snapshot, and detail loading must not execute skill scripts; any execution affordance must remain default-denied and require a future per-request preview/confirmation/audit path

Smoke App Run must not use the real user Claude config.
It must also avoid real user Codex and opencode config; fixture mode uses temporary `HOME`, app data, and project roots only.
Smoke and real-local screenshots must not expose the actual local `HOME`, app data, `/var/folders`, temporary fixture roots, tokens, or credential placeholders. Redact any visible paths before committing evidence, and run `pnpm check:privacy`; new screenshots still require manual visual inspection.

## V2.7 LLM Local Assist Validation

V2.7 validation must treat LLM as default-off local assist. The current implementation boundary is service/UI gate plus request prepare/estimate; it must not require or use real provider credentials.

Automated validation should cover:

- app launch performs no LLM provider network call
- Analyze / Recommend / conflict explanation / draft frontmatter actions require a user-triggered prepare/estimate path
- prepare/estimate displays provider, model, estimated input/output tokens, estimated cost, budget state, and disabled/unconfigured reason
- missing provider or disabled LLM state blocks execution before any provider client can be created
- credentials are not written to SQLite, project directories, logs, smoke fixtures, or app-window screenshots
- draft frontmatter is display/copy-only; no Apply / Write control is exposed from LLM output

Fixture smoke must use temporary `HOME`, app data, and project roots only. It must not read, create, or modify real user LLM credential files. If a future fallback credential fixture is required, it must live under the temporary fixture home and assert `0600` permission behavior.

Real local validation must verify the UI communicates disabled/default-off LLM state without prompting for or saving a real API key. Do not enter real provider credentials during smoke or docs closeout. The 2026-06-09 current mainline pass verified the disabled controls; future candidates should repeat this check when LLM UI changes.

2026-06-08 V2.7 closeout record:

## V2.10 Skill Execution Safety Validation

V2.10 validation must treat skill execution as a default-deny boundary. It must not claim a real script runner, sandbox runtime, successful execution output log, public execution API, or app-store-style release artifact exists.

Automated validation and docs closeout should cover:

- protocol v1 is default-deny for execution: `script.previewExecution` and `script.execute` are supported for safe preflight and confirmation-gated attempt flow, while real script execution is deferred until a sandbox runner is shipped.
- scan, import, export, install, LLM prepare, state snapshot, detail loading, and smoke fixture flows do not spawn skill scripts or helper commands from skill content
- any future execution affordance requires per-request user confirmation and shows cwd/env/network/files preview before execution can be considered
- blocked, cancelled, and failed attempts have an audit record contract; `completed` execution records remain invalid until a real sandbox runner exists
- LLM output cannot trigger execution, set confirmation, or bridge into execution-like service calls

The current mainline app passed real local Computer Use on 2026-06-09. The pass verified that the visible UI does not present script execution as a completed working feature; the real catalog had no structured script command records, so the preview reached a safe missing-command preview-only state without execution. If an execution affordance changes in a future build, verify it is default-denied, preview-first, confirmation-gated, and records blocked/cancelled/failure attempts without running the script unless the real sandbox runner has separately shipped. Fixture smoke screenshots and launch verification do not close this real local validation item for future candidates.

2026-06-09 V2.10 docs/release closeout:

- Documentation was updated to describe the safe boundary: no real execution by default, per-request confirmation, cwd/env/network/files preview, audit records for blocked/cancelled/failure attempts, and no LLM-triggered execution.
- Public release/signing/notarization/DMG automation remains deferred.
- Real local Computer Use passed for the current mainline app on 2026-06-09; future candidates must rerun it and keep any blocker explicit.

## Screenshot Rule

Completed UI screenshots must be app-window-only captures. Use:

```sh
pnpm capture:macos-window
```

or:

```sh
script/capture_app_window.sh SkillsCopilot docs/ui-artifacts/native-macos-shell/completed.png
```

Full desktop screenshots are forbidden.
Visible local paths, usernames, app data directories, temp directories, tokens, and credential placeholders are also forbidden in committed screenshots. Use redacted screenshots for repository artifacts and keep raw local captures out of git.

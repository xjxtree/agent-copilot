## Bounded Cockpit Loading

The implementation uses bounded local aggregation metadata across task and remediation paths:

- `task.checkReadiness` records candidate scan limits, scanned/total counts, completed/skipped stages, blocker codes, and notes.
- `task.rankSkillRoutes` reuses the bounded readiness result and appends routing metadata instead of recomputing heavy paths.
- `task.compareAgentReadiness` bounds benchmark context and reports aggregation metadata.
- `remediation.plan` bounds detail scans and records overflow as partial metadata.
- `remediation.previewDrafts` and `remediation.previewImpact` remain read-only preview helpers and are skipped by batch review when the bounded budget is exhausted.
- `remediation.batchReview` returns partial plan/cleanup rows when draft/impact previews would exceed budget.
- `task.buildCockpit` has a default 15s UI budget, service-side timeout metadata, optional context skipping, fallback reasons, and top-level `partial` / `elapsed_ms` fields.

The verifier should not accept a single unbounded spinner, indefinite `Preparing...`, or a hidden background task as future completion evidence.

## Timeout, Fallback, Cancel, Retry

Closeout evidence covers these cases:

- Slow remediation context returns a bounded fallback row without blocking all Cockpit output.
- Cancel stops the visible request and prevents stale completion from overwriting the cancelled state.
- Retry runs the same task again from the visible Task Cockpit UI.
- Failure/fallback rows preserve gap notes, blocker notes, evidence refs, prompt metadata, and safety flags where available.
- Timeout and retry state remains read-only and does not trigger provider confirmation, write preview, apply, snapshot, rollback, or script execution.
